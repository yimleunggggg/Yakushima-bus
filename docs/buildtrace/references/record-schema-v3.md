# BuildTrace record schema v3

v3 在 v2 的证据链上增加三类可选结构：复盘轨、跨项目决策原则和发布可见范围。`docs/buildtrace/BUILDTRACE.md` 仍是唯一可编辑主源；JSON、个人决策手册、团队快照和公开快照都只是可重新生成的派生物。

## 安全默认值

每条记录可以声明：

```md
可见范围: private | team | public
```

- `private`：完整私密记录，可包含原话、内部路径、Agent 过程和项目入口。旧记录没有此字段时一律按 `private` 处理。
- `team`：允许进入团队快照。生成时仍移除私密记录、任务来源、项目入口、绝对路径和本地链接；团队快照必须放在有访问控制的环境中。
- `public`：允许进入脱敏公开快照。生成时还会移除沟通原文、Agent 内部过程和技术备注。

公开分享必须运行 `buildtrace publish --profile public` 生成新目录。只在浏览器里隐藏内容不是安全边界，因为原始 Markdown 仍可能被下载。

Agent 只能在用户明确表达共享意图时把记录标为 `team` 或 `public`；不能根据内容“看起来不敏感”自动公开。

## 复盘轨

没有对应内容时可以省略这些可选小节。不要为了填满结构而发明假设、指标或结果。

```md
人的假设:
- [H1] 如果把人的判断和动作放在同一条记录里，之后的数据复盘会更容易归因。

观察计划:
- [W1 | 指标=回看完成率 | 数据源=项目记录 | 基线=未记录 | 目标=连续 3 次完成 | 窗口=30 天 | 达成规则=到期后有数据和明确判断] 验证这条复盘轨是否真的被使用。

数据变化:
- [D1 | 日期=2026-08-18 | 指标=回看完成率 | 值=3/3 | 变化=从未记录到连续完成 | 来源=项目记录] 三条到期记录均补上了后续结果。

结果判断:
- [O1 | supported | 日期=2026-08-18] 观察期内成立，继续使用。
```

结果判断的推荐标签是 `supported`、`refuted`、`mixed`、`inconclusive` 或 `pending`。它描述某条假设后来是否成立，不替代记录级的 `结果状态`。

同一条记录可以先只有假设和观察计划，后来再追加数据变化与结果判断。必须保留前后时间与证据，不能用后来认识重写最初假设。

## 决策原则

真正需要跨项目复用的判断使用稳定 ID：

```md
决策原则:
- [dp-human-judgment-first | 产品] 工具降低实现成本时，更要保留人的判断、假设和取舍。
```

- ID 必须是 `dp-` 加小写英文、数字或连字符。
- 同一条原则在不同项目里沿用同一个 ID，标题或表述微调时不要随意换 ID。
- 只有真实记录明确支持时才添加原则，不要从相似文案自动推断。
- `buildtrace playbook --target <project-a> --target <project-b>` 按 ID 聚合证据。
- 同一 ID 出现不同表述时标记为冲突，交给人确认；程序不能擅自融合。

## 可迁移派生物

- `buildtrace export --format json|jsonl`：完整私密结构化导出，schemaVersion 为 3。
- `buildtrace playbook`：跨项目原则索引，证据仍链接到各项目记录。
- `buildtrace publish --profile team|public`：从主源生成一个新的、只读、已脱敏静态目录。

派生物不能反向覆盖主源，也不能被 Viewer 或 Agent 当成第二份项目事实。
