# BuildTrace Agent 工作流

读取时机：当用户要求记录本轮、恢复上下文、复盘时间段、补关键原因、纠正/隐藏/删除/备注节点、设计 hooks/rules/memory，或评价 BuildTrace 是否太像手填表单时读取本文件。

## 核心原则

BuildTrace 面向 vibe coding 用户。用户通常已经在和强 Agent 协作，所以 BuildTrace 不应该变成一个要求用户手填字段的表单。

正确分工：

- Agent 负责观察、起草、归档、引用证据、标注不确定性。
- 用户负责确认真实原因、纠错、补后台数据观察、给出结果判断，或给某条记录加一句自己的备注。

## 四个主要入口

### 恢复上下文

用户说“继续”“恢复上下文”“看一下项目状态”时：

1. 读 `PROJECT-PROFILE.md`。
2. 读最近 3-5 个节点。
3. 读待回看和待补原因节点。
4. 用 5-8 句话告诉用户现在项目在哪里、有哪些未闭环事项。

不要一次性加载全部历史，除非用户正在做历史复盘。

### 记录本轮

任务完成后，Agent 自动起草节点。

起草内容：

- 本轮用户原话或关键判断。
- 人话摘要。
- 加了、改了、减去了、优化了什么。
- 技术备注。
- 依据/来源，例如 commit、文件、截图路径、URL、数据导出。
- 建议回看信号。

用户只需要确认、纠错、补一句真实动机。

### 复盘时间段

用户看流量、交易、事件、部署或错误时：

1. 读取对应日期范围内的节点。
2. 读取外部事件或提示用户导入/手填。
3. 按日期把项目节点和外部事实并排列出。
4. 明确哪些只是同段时间发生，哪些有证据支持因果。
5. 更新 `review.result` 和 `review.learning`。

### 补关键原因

只问高价值缺口，不问所有缺口。

优先问：

- 影响数据复盘的节点。
- 影响产品方向的节点。
- 影响变现、交易、SEO、架构、安全的节点。
- Agent 从证据无法可靠推断的节点。

问题应该短：

```text
这个节点我只能从文件推断结果，但不知道你当时为什么这样做。真实动机更接近哪一句？
```

## 意图收件箱

对话过程中不要频繁写文件。Agent 应在心里暂存高价值信号：

- “我想……”
- “我担心……”
- “不要做……”
- “先不做……因为……”
- “如果有效应该看到……”
- “这个参考给了我启发……”
- “这只是临时方案……”

任务结束后再压缩进节点。

## 用户纠正、隐藏、备注

用户说“这条不需要”“这个写错了”“这个日期不对”“我想补一句备注”时：

1. 不要争辩，也不要默认删除。
2. 如果是事实错误，修正节点，并在 `userNotes` 写明用户纠正了什么。
3. 如果用户只是不想在看板看到，设置 `hidden: true`。
4. 如果用户明确要求删除，先备份 `nodes.json`，再删除。
5. 如果用户只是想留一句主观判断，写入 `userNotes`，不要强行改成 `review.learning`。

结果回看不是强制任务。用户可以把某条标记为 `not-needed` 或只保留为历史记录。

## Hooks / Rules / Memory 的用法

通用做法：

- 项目规则：告诉 Agent 什么时候读 BuildTrace，什么时候写节点。
- 会话开始：如果是旧项目，先恢复上下文。
- 任务结束：若发生有意义变化，自动起草节点。
- 提交/部署前：提醒截图、证据、回看信号。
- 数据复盘时：读取节点和外部事件，帮助用户判断结果。

不要强依赖某个工具。Cursor、Codex、Claude Code、GitHub Copilot、Trae 或其他 Agent 工具都可以通过规则/说明文件接入类似流程。

## 不要做的事

- 不要要求用户在开发中途填完整表。
- 不要把所有 commit 都变成节点。
- 不要把外部数据自动解释成因果。
- 不要编造聊天记录里没有的原话。
- 不要为了时间线完整而牺牲证据可信度。

## 好节点的标准

三周后用户还能用它回答：

- 当时为什么做？
- 实际改了什么？
- 技术上怎么做？
- 证据在哪里？
- 后来有没有用？
- 如果不确定，为什么不确定？
