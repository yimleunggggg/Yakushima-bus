# BuildTrace 场景字段

读取时机：当用户要求记录节点、回填旧项目、修复节点质量、设计看板字段，或需要判断节点类型时读取本文件。

## 通用字段

每个节点都需要：

- 场景：这个节点主要属于什么工作类型。
- 人话摘要：非技术背景的人能一眼看懂发生了什么。
- 技术备注：给开发者看的实现、文件、结构、命令、风险。
- 证据：来自哪个文件、commit、截图、聊天原文或数据导出。
- 日期可信度：这个日期是精确证据、从 commit 推断、从文档推断，还是不确定。

## 使用时机

BuildTrace 不应该在开发过程中不停让用户填表。

适合使用：

- 开工前：恢复上下文，读项目档案和最近节点。
- 重要动作结束后：用 1 个节点压缩本轮意图、变化、证据、待看结果。
- 复盘数据时：把项目节点和外部时间线叠起来，看同一时间段发生了什么。
- 回填旧项目时：先做证据覆盖报告，再写少量高可信节点。
- 用户说“这个要记住”“为什么当时做这个”“以后复盘要看”时。

不适合使用：

- 每个小代码编辑后都打断用户。
- 纯格式、纯重命名、纯依赖整理且未来不影响复盘。
- 没有项目语境的闲聊。
- 证据不足却为了时间线完整而硬编节点。

## 人话与技术分层

每个节点外层只讲人能扫懂的事：

- 标题：对象 + 动作/结果。
- 人话摘要：这件事发生了什么，为什么重要。
- 场景细节：问题、方案、成功信号、风险。

技术内容放进技术备注：

- 文件、命令、数据结构、架构、验证方式。
- commit、PR、部署、API、配置。
- 给未来 Agent 或工程师复现用的信息。

不要把标题写成 `CLS pass / next-bar / dashboard` 这类只有开发者知道的词。

## 场景类型

### 功能新增 feature

记录：

- 用户/使用场景
- 新增能力
- 成功标准
- 入口、页面、流程或 API
- 是否需要后续看数据

### Bug 修复 bugfix

记录：

- 问题现象：用户看到什么坏了
- 影响范围：谁受影响、影响多大
- 根因：已确认、推断、未知
- 解决方案：这一轮具体修了什么
- 验证方式：复现步骤、测试、截图、日志、线上检查
- 残余风险：是否可能复发

### 优化 optimization

记录：

- 优化目标：更快、更清楚、更稳、更好转化、更易维护
- 改前摩擦
- 改后变化
- 对比证据
- 后续观察指标

### 实验 experiment

记录：

- 假设
- 目标人群
- 变量/版本
- 成功指标
- 观察周期
- 结果与学习

### 数据/埋点 analytics-data

记录：

- 要回答的问题
- 事件/字段/指标定义
- 数据来源
- 校验方式
- 数据污染或口径风险

### SEO/增长 growth-seo

记录：

- 搜索意图或增长假设
- 页面/关键词/渠道
- 改动内容
- 观察指标：曝光、点击、CTR、排名、关键事件
- 结果归因限制

### 变现/交易 monetization

记录：

- 商业假设
- 触发场景
- 商品/价格/联盟/支付/订阅
- 转化路径
- 风险：误导、过度打扰、合规、退款、支付失败

### 架构/重构 architecture

记录：

- 为什么现在要改
- 旧结构的问题
- 新结构的边界
- 迁移方式
- 回滚方式
- 长期维护收益

### 内容/文案 content

记录：

- 目标读者
- 原内容的问题
- 新表达策略
- 影响页面/渠道
- 需要观察的行为或反馈

### 运维/发布 ops-release

记录：

- 发布内容
- 环境/配置
- 验证清单
- 回滚方式
- 事故或异常

### 移动 App app-mobile

记录：

- 平台和版本
- 设备/系统范围
- 权限、通知、离线、性能、商店审核风险
- 复现设备和验证设备

### AI/提示词/模型 ai-prompt

记录：

- 任务目标
- 模型/提示词/工具变化
- 输入输出边界
- 失败样例
- 评估方式

### 安全/隐私 security-privacy

记录：

- 风险类型
- 涉及数据
- 权限边界
- 修复或缓解方式
- 验证和残余风险

## 日期与证据规则

- 如果日期来自 commit，写 `日期可信度: commit-exact`。
- 如果日期来自文档标题或日志块，写 `日期可信度: doc-exact`。
- 如果只是按文件修改时间或上下文推测，写 `date-inferred`，并在标题或摘要里避免绝对化。
- 如果无法确认，不要填具体日期，写 `unknown` 或请求用户确认。
- 不要把“某天仓库初始化”扩写成“某功能某天上线”，除非证据明确支持。

## 标题规则

标题应该让人在外层 3 秒内看懂：

- 用中文短句。
- 12-24 个字左右。
- 包含对象 + 动作/结果。
- 避免只写技术名词，例如 `next-bar`、`dashboard`、`CLS pass`。
- 技术词可以放在技术备注，不要放在主标题里，除非它就是用户熟悉的核心对象。
