# Trace / BuildTrace 中文发布文案草稿

我最近在想一个 AI 开发里越来越明显的问题：

东西改得很快，但“为什么这么改”很容易消失。

Git 能告诉我哪些文件变了。Changelog 能告诉我发布了什么。数据后台能告诉我某天流量或转化变了。但过了几天或几周，我常常想不起来：

- 当时为什么要做这个？
- 我在担心什么？
- 我参考了什么？
- 我期待它影响哪个数据？
- 当时页面长什么样？
- 技术上到底怎么实现的？
- 后来它到底有没有用？

这些信息很多都在和 AI 的对话里。可一旦对话被压缩、归档、换工具、换线程，人的意图和判断就很容易沉下去。

所以我在做一个小东西：Trace。

对外看板叫 Trace，Skill / 包名叫 buildtrace。

它是一个给 AI 协作开发用的项目节点记忆系统。

它不是传统 changelog，也不是项目管理工具。它想记录的是一个项目里每个重要节点：

- 人当时为什么这样想
- 项目实际增加、修改、减少、优化了什么
- 技术上用了什么实现方式
- 当时有哪些截图、链接、数据基线或参考
- 后续应该看什么结果
- 结果回看时到底有效、无效，还是不确定
- 同一时间段里还有哪些外部事实：部署、PR、GA4/PostHog 事件、Vercel 上线、Stripe 交易，或用户手填的数据观察

我最初是从运营视角想到这个需求的。因为看数据复盘时，最难的不是知道“数据变了”，而是想起来“那段时间我到底做过什么，为什么做，哪些事情可能影响了它”。

现在 AI 让更多非程序员也可以开发产品、小工具、网页、内容站、电商页和 App。代码越来越容易生成，但“当时为什么做”、判断和取舍反而更需要被保留下来。

它不会在开发过程中逼你不断填表。更合理的使用方式是让 Agent 先做草稿，人只补关键判断：

- 开工前：恢复上下文，读最近节点和未完成判断
- 重要工作结束后：记录本轮，把改动、意图、证据沉淀成一个节点
- 看数据时：复盘一个时间段，把 GA4、GitHub、Vercel、PostHog、Stripe 或手填观察叠到项目时间线上
- 历史不清时：生成“意图收件箱”，只问那些真正重要但缺人的判断的问题

Trace / BuildTrace 的第一版会包括：

- 给 AI agent 用的 Skill / 指令
- 项目节点记录模板
- 项目基础信息模板
- 旧项目回溯建档流程
- 一个本地前端看板，用来浏览节点、证据、缺口和待回填结果
- 外部时间线的导入/手填规范，用来把 GA4、GitHub、Vercel、PostHog、Stripe 等事实叠到项目节点旁边

我想先把它做成一个能真实使用的小工具，而不是只停留在文档规范。

视觉上，我不想做成那种普通 SaaS 后台，也不想做成米色 AI 模板。更接近一个冷感科技手账：清晰、克制、可回溯，但仍然保留人的原话和当时判断。

因为未来复盘时真正有价值的，不只是“我做了什么”，而是：

我当时为什么相信这件事值得做。
