## Yakushima-bus 项目 Token 估算（会话级）

> 口径说明：本文件以 **v2（按 content.type 精分）** 为准。  
> 早先 v1 结果（输入 `8.9M-19.4M`、输出 `0.37M`）因分类口径较粗，已废弃。

- 统计范围：`agent-transcripts` 下父会话（不含 subagents），共 `11` 个会话文件。
- 记录总数：`7017`；其中用户消息 `1046` 条、助手消息 `5971` 条。
- 估算方法：按中英混合字符做粗略换算（中文约 `0.9 token/字`，非中文约 `4 字符/token`），仅用于趋势判断，不等于官方账单。

### 当前估算结果（已按会话重算）

- 用户文本：约 `485,073` tokens
- 助手文本（可见输出）：约 `245,256` tokens
- 可见输入（用户文本 + tool_use 指令体 + 少量历史重放近似）：约 `2,646,882` tokens
- 隐式上下文（系统/规则/skills 列表/环境注入，按每轮 `8k-18k` 估算）：约 `8,368,000` 到 `18,828,000` tokens
- 项目累计输入总量区间：约 `11,014,882` 到 `21,474,882` tokens

### 为什么和第一次不一样（v1 -> v2）

- v1 把很多 assistant 消息整体按字符串估算，未把 `tool_use` 与 `text` 严格拆开，导致“输出”偏高、可见输入偏低。
- v2 按 `message.content[].type` 拆分：`text` 计入用户/助手文本，`tool_use` 单独统计为工具相关可见输入。
- 因此 v2 会出现：助手可见输出下降（`0.37M -> 0.245M`），但总输入上升（`8.9M-19.4M -> 11.0M-21.5M`）。
- 两版都不是官方账单，只是工程估算；v2 的结构化口径更稳定，后续都应沿用 v2。

### 结构判断

- 最大头是每轮固定注入的上下文，不是用户提问本身。
- `skills` 的主要成本来自“skills 列表和规则被注入上下文”，不是“调用某个 skill”这件事本身。
- 工具调用越多，模型读写的中间内容越多，但在该项目里仍通常小于固定上下文成本。

### 结论补充

- 前四个长会话基本吃掉了绝大多数 token，总成本和“用户轮次”强相关。
- 工具密度高会抬高可见输入，但总体上仍被“每轮固定上下文”主导。
- 若只想快速降耗，优先把长会话拆短，并在新会话减少不必要的工具往返。

### 下一步建议

- 若要降耗，优先减少每轮注入体量（精简规则/技能暴露范围）和无效轮次。
- 建议后续做“按会话分组”的趋势看板（每个会话输入/输出区间 + 轮次 + 工具调用密度）。
